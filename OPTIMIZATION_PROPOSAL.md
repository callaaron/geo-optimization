# GEO 优化系统 — 四维优化方案

> 基于当前代码架构（`/app` 目录）全面审计后制定。所有改动不破坏现有 AI 链路（DeepSeek v4-pro / Ark Agent Plan），前后端可独立迭代。

---

## 一、AI 检索工具明确化 + 外部内容评分体系

### 现状
- 搜索：仅 360（主）+ Bing（备），`search.mjs` 零依赖 HTML 解析
- 评分：`relevanceScore()` 用 2-3 字 n-gram 重叠率做 L1 近似，存在较明显的数量偏差（长文本得分偏高）
- 无内容权威度、时效性、发布平台权重的概念
- AI 认知检测（`aiCitation`）只有"提及/未提及"二元判断

### 优化方案

#### 1.1 检索工具矩阵（分层，按场景自动选择）

| 层级 | 工具 | 用途 | 接入方式 |
|------|------|------|---------|
| **L1 网页搜索** | 360 搜索 (已有) | 中文 B2B 高精度 SERP | 保持 `/search.mjs` |
| **L1 网页搜索** | Bing (已有) | 备用引擎 | 保持 |
| **L2 信源内容** | 网页正文抓取 (`fetchPageText`) | 深度内容分析 | 已有，增强超时/编码检测 |
| **L2 信源内容** | ⭐ **CORS 代理多端点** | 绕过反爬 | 新增 `fetch.ts` 聚合 |
| **L3 AI 引用** | DeepSeek RAG 综合 (已有) | 模拟 AI 问答中的品牌可见度 | 保持 `chat()` + RAG prompt |

不新增外部付费搜索 API（保持零边际成本），但把已有工具**命名清楚、分层调用**。

#### 1.2 外部内容评分体系（替代当前 n-gram 近似）

新增 `lib/geo/scorer.ts`，对每条信源计算 5 个维度分（0-100）：

| 维度 | 指标 | 计算方式 |
|------|------|---------|
| **相关性** | 与 query 的语义匹配度 | 基于关键词 TF + 实体重叠（保留 n-gram 但加 Jensen-Shannon 归一化） |
| **权威度** | 发布平台可信度 | 域名白名单（.gov.cn/.edu.cn/主流门户=30分）+ 行业垂直媒体=20分 + 个人站点=5分 |
| **时效性** | 内容新鲜度 | 页面内有日期的解析距今天数（<1月=30分, <1年=20分, >1年=5分, 无日期=0分） |
| **完整度** | 信息密度 | 字数 > 500 = 10分；含结构化数据(表格/列表/Schema) = 10分；总计 max 20分 |
| **可引用性** | 是否含可被 AI 直接引用的片段 | 含实体定义/数据锚点/FAQ/element编号 = 额外 10分 |

**总分 = 加权求和**（相关性 x0.4 + 权威度 x0.2 + 时效性 x0.15 + 完整度 x0.1 + 可引用性 x0.15），分档：

- 🟢 **高质量信源** ≥70 分 → 优先引用
- 🟡 **中等** 40-69 → 可引用但标注
- 🔴 **低质量** <40 → 仅作参考，不进入 AI 综合上下文

#### 1.3 AI 认知检测升级（从二元到分级）

`aiCitation` 当前输出 `mentioned: true/false`。升级为 4 级：

| 级别 | 含义 | 判定条件 |
|------|------|---------|
| **直接引用** | AI 明确提到品牌/域名并给出具体信息 | answer 含品牌名 + 至少 1 个实体属性 |
| **间接提及** | AI 提到了相关品类/话题，但未指名 | prompt 已有但 answer 回避了品牌名 |
| **可触发提及** | 当前未提，但补充 X 后即可 | judge.reason 分析出只需补某个信息点 |
| **未提及** | 完全不在 AI 认知范围内 | 既无品牌名也无相关品类描述 |

## 二、内容输出策略（系统性工程规划）

### 现状
- 输出物：HTML 审计报告 + CSV 导出 + Markdown 改写 + llms.txt/JSON-LD/robots.txt
- **没有图文、视频、信源网站的输出方案**
- 改写器只做"AI 友好 Markdown"，没有"这个内容应该发在哪、用什么格式"的策略层

### 优化方案

#### 2.1 输出物矩阵（按 GEO 目标分层）

```
用户输入 → 系统分析 → 输出策略引擎 → 多格式输出
                                    ├── 📄 图文内容（公众号/知乎/小红书）
                                    ├── 🎬 视频脚本（抖音/视频号/B站）
                                    ├── 🌐 信源网页（官网落地页 / llms.txt）
                                    ├── 📊 数据报告（审计报告 HTML/PDF）
                                    └── 🔗 结构化数据（JSON-LD / Schema / FAQ）
```

#### 2.2 新增输出能力

| 输出类型 | 实现方式 | 触发条件 |
|---------|---------|---------|
| **公众号/知乎图文** | LLM 生成（标题+导语+小标题分段+CTA），前端即时渲染预览 | 内容改写：选"图文模式" |
| **小红书笔记** | 短文案+emoji+标签+配图建议（不生成图，给 prompt） | 内容改写：选"社媒模式" |
| **视频口播脚本** | LLM 生成 60s/3min 两种长度的口播稿+分镜建议 | 新增"视频脚本" Tab 或融入改写器 |
| **官网落地页** | 基于审计结果自动生成页面结构（H1/段落/FAQ/Schema） | 项目审计完成后一键生成 |
| **llms.txt 增强** | 已有，增加按内容类型分类输出（产品页/文章/FAQ 分块） | 生成器增强 |

#### 2.3 策略引擎（`lib/geo/strategy.ts` 新增）

输入：品牌类型(B2B/B2C)、目标平台、竞品分析结果
输出：`{ format, channel, priority, template }` 的推荐清单

```typescript
interface OutputStrategy {
  format: "article" | "social" | "video" | "landing" | "structured_data"
  channel: string        // "公众号" | "知乎" | "小红书" | "官网" | "抖音"
  priority: 1 | 2 | 3   // 1=立即产出 2=近期 3=可选
  template: string       // prompt 模板引用
  reason: string         // 为什么推荐这个（基于竞品缺口）
}
```

## 三、前端页面设计重构

### 现状问题
- 7 个 Tab 全部平铺，视觉拥挤，功能重叠（监控台和数据看板；分析器和流程审计）
- 深色主题不可切换（虽是设计风格但缺少日间模式）
- 表单项过多，布局线性单调，信息层级不清晰
- 结果区域（监测报告/分析结果）渲染在表单下方，长页面滚动疲劳

### 优化方案

#### 3.1 Tab 合并精简：7 → 4

```
现有                              优化后
────────────────────────────────────────────
监控台 (dashboard)        ─┐
数据看板 (monitor)         ├──→ 🏠 仪表盘 (Dashboard)
                          │    左：项目列表+状态卡片
                          │    右：监测结果+趋势图
                          │
GEO 服务流程 (workflow)    ─┐
分析评分器 (analyzer)       ├──→ 🔍 分析与审计 (Audit)
llms.txt 生成 (generator)   │    顶：输入区（见"智能输入"优化）
                          │    中：分析结果+评分
                          │    底：一键生成 llms.txt/报告
                          │
内容改写 (rewriter)        ───→ ✏️ 内容工厂 (Content)
竞品对标 (benchmark)       │    左栏：改写+多格式输出
                          │    右栏：竞品对比+策略建议
                          │
（新）                    ───→ 📁 项目管理 (Projects)
                               项目列表/历史审计/导出
```

#### 3.2 视觉升级

| 项目 | 当前 | 改为 |
|------|------|------|
| 主题 | 仅深色 | 深色/浅色切换（`next-themes` 已有，加 toggle） |
| 布局 | 单列全宽 | 左右分栏（输入左 + 结果右），支持折叠 |
| 卡片 | 扁平，无阴影层级 | 增加 elevation（shadow-sm/md/lg 适当分层） |
| 数据展示 | 裸数字+BarChart | 增加 KPI 卡片（数值+环比+趋势小火花图） |
| 空状态 | 空白或占位文字 | 增加插画式 empty state + 引导按钮 |
| 配色 | 仅有功能色 | 增加 GEO 品牌色系统（主色/辅色/语义色/中性色） |
| 移动端 | 无响应式 | 768px 断点：Tab 折叠为汉堡菜单，结果区叠放 |

## 四、智能输入（减少填写量 + 多模态）

### 现状
- 所有 Tab 均要求用户手动逐字段填写（品牌名、域名、query、内容点……）
- 仅支持粘贴文本和输入 URL 两种输入方式
- 无文件上传、无自动提取、无智能补全
- 竞品、query、内容点等"衍生字段"完全依赖用户自行思考填写

### 优化方案

#### 4.1 输入方式扩展

| 输入方式 | 前端实现 | 后端支持 | 适用场景 |
|---------|---------|---------|---------|
| 📝 手动填写 | 保持现有表单 | - | 精确控制 |
| 📋 粘贴文字 | 保持现有 | - | 已有文案 |
| 🔗 输入 URL | 保持 CORS 抓取 | - | 分析竞品页面 |
| 📎 **上传文件** | 拖拽/点击上传 | 新：`/api/upload` → 文件解析 → LLM 提取 | PPT/Word/PDF/图片 |
| 🎤 **语音输入** | Web Speech API (浏览器原生) | - | 快速口述需求 |
| 🤖 **一键智能提取** | 调用后端 LLM API | 新：`/api/ai/extract-profile` | 粘贴公司简介 → 自动生成品牌名/域名/行业/query/内容点 |

#### 4.2 文件上传管道

```
用户上传 PPT/Word/PDF/图片
        ↓
POST /api/upload (multipart/form-data)
        ↓
后端解析：
  - PDF → pdf-parse (纯文本提取)
  - Word → mammoth (.docx → HTML → 纯文本)  
  - PPT → pptx2json (提取文本框内容)
  - 图片 → 暂不做 OCR（iOS/Android 截图可通过系统分享文字）
        ↓
文本 → LLM (/api/ai/extract-profile) → 结构化 JSON
        ↓
返回：{ brand, domain, industry, queries[], competitors[], content_points[] }
        ↓
前端自动填充所有表单字段
```

#### 4.3 智能补全链（输入一个，联想一串）

用户只需填**品牌名**，系统自动：

```
品牌名 → AI 分析 → 
  ├── 推断行业
  ├── 生成行业 query（基于 LLM 对行业的理解）
  ├── 推荐竞品（让 LLM 列出该行业的知名品牌）
  └── 建议内容点（基于行业常见话题）
```

实现：新增 `/api/ai/suggest` 端点，输入 `{ brand, industry? }` → 输出 `{ queries[], competitors[], contentPoints[] }`

#### 4.4 简化后的用户流程（对比）

**现在（以监控台为例，需手动填 5 个独立字段）：**
```
1. 品牌名 *      → 用户手动输入
2. 域名（可选）   → 用户手动输入
3. 竞品（可选）   → 用户手动思考+输入
4. 监测 query *  → 用户手动思考+输入
5. 内容点（可选） → 用户手动思考+输入
然后点"运行监测"
```

**优化后（两步完成）：**
```
1. 品牌名 *  → 用户输入（或粘贴公司简介/上传文件自动提取）
2. 点「智能补全」→ AI 自动填充：域名/行业/竞品/query/内容点
   （用户可微调，也可直接接受）
然后点"运行监测"
```
