## 3. AI 增强诊断（模块① analyze，真实大模型）

**总结**：内容具备基础企业信息与资质编号，但缺乏产品参数、案例与转化细节，当前被 AI 引擎引用的可能性**中等偏低**，主因是 B2B 决策所需的规格化数据与实证不足。

**优势**：食品生产许可证（SC10437028204871）、「岛城放心消费品牌」2026 背书可作为信任信号；产品线（鲅鱼/黄花鱼/墨鱼水饺）与 OEM/批发模式清晰；企业主体信息完整（成立时间、即墨区、控股「饺大师」）。

**缺口（按影响）**：
1. 缺产品规格参数（单只克重、每袋规格、冷链温度、保质期、含肉率、MOQ、交货周期）。
2. 无客户案例与实证（合作连锁/商超、供货量、复购数据）。
3. 转化路径模糊（仅「联系我们」，无电话/邮箱/微信/地址/询价表单/样品流程）。
4. 缺选型对比与差异化量化数据（「手工包制」「锁鲜速冻」为定性描述）。

**前 5 条改进动作**：增设产品规格表 → 增加合作案例模块 → 补充联系与询价区块 → 新增「为什么选正岛」对比段 → 开头加 FAQ 式定义句。

> 规则引擎（overall 40/E）与 AI 诊断结论**高度一致**：均指出缺规格参数、缺客户实证、转化模糊、信任/资质不足。两路管线结论对齐。

## 4. 内容改写（模块②：规则 vs AI）

### 4.1 规则改写引擎（rewriter）

可用，但质量有限。产出 TL;DR + 关键数据 + 1 条定义 + 1 条 FAQ + 正文。**暴露 3 处缺陷（待模块侧修复，见第 8 节 P2）**：
- 定义抽取误判：把「位于山东青岛**即**墨区」中的地名接续字「即」误识别为释义标记，抽出错误定义 `位于山东青岛 → 墨区，是……`。
- FAQ 无信息量：把整段标题/首句当成问句（「……是什么？」），答案=原句复述。
- TL;DR 与首句完全重复，未做压缩提炼。

### 4.2 真实 AI 改写（rewrite，走后端 `:8787`）

**修复后实测返回 `ok:true`**，输出：TL;DR 1 条 + FAQ **7** 条 + 定义 **7** 条 + 改进点 **5** 条。示例：
- TL;DR：`青岛正岛食品：海鲜水饺与冷冻鱼糜制品专业生产商，支持贴牌代工与批发。`
- FAQ 覆盖：是否支持 OEM/ODM、生产资质、主要产品、原料工艺、品牌背书、如何获取报价、公司地址。
- **未编造**：对原文未提供的联系方式、具体规格（克重/箱规）、详细地址，一律用 `【待补充：…】` 占位——符合 GEO「不编造」要求。

### 4.3 规则 vs AI 对比

| 维度 | 规则改写 | AI 改写（修复后） |
|------|---------|----------------|
| 结构 | 平铺，仅 TL;DR+数据+1 定义+正文 | TL;DR + 关键要点 + 6 层正文 + FAQ + 询价路径 |
| FAQ | 1 条，标题复述无信息量 | 7 条，B2B 导向 |
| 定义/实体 | 1 条且**误判** | 7 条准确（公司/许可证/海鲜水饺/OEM/荣誉/关联企业） |
| 缺失数据 | 无处理 | 用 `【待补充】` 占位 |
| 适合落地 | 零成本兜底骨架 | **更适合正岛直接落地** |

## 8. 测试中发现的缺陷与修复（关键价值）

### P0 — AI 改写接口整模块失效（已修复 ✅）

- **现象**：`/api/ai/rewrite` 连续 5/5 返回 `{"ok":false,"error":"模型未返回可解析的 JSON"}`；同 Key、同模型、同路径的 `/api/ai/analyze` 正常 → 说明是 rewrite 接口特有问题。
- **初步误判（红鲱鱼）**：曾怀疑 `maxTokens: 2600` 截断。但实测模型输出仅 **~2784 字符（远低于 4096）且以 `}` 正常结尾**，排除 token 不足。
- **真实根因**：`ark-code-latest` 模型在 JSON 字符串值内部写入了**未转义的真实双引号**——如中文文案 `2026年入选"岛城放心消费品牌"\n`，其中 `"` 是真实 ASCII 双引号（char code 34），位于本就由 `"` 界定的 JSON 字符串值内部，导致 JSON.parse 把第一个 `"` 当成字符串结束符而解析失败（`Expected ',' or '}' after property value at position 407`）。
- **修复**：在 `server/ark.mjs` 的 `parseJsonLoose` 中新增 `repairJson` 修复器——
  - 用「看引号后下一个非空白字符是否为 JSON 分隔符（`,` `}` `]` `:`）」的启发式，区分字符串边界引号与内容引号，内容引号自动转义为 `\"`；
  - 字符串值内的真实换行/制表符转义为 `\n`/`\t`；
  - 遍历结束仍在字符串内则补闭合引号（截断兜底）。
  - 解析顺序：严格解析 → 截到末括号 → 修复后再解析。
- **验证（修复后真实调用）**：`/api/ai/rewrite` 返回 `ok:true`，7 FAQ / 7 定义 / 5 改进点，无编造。`analyze` 与 `citation` 接口同步回归正常（`ok:true`），修复未引发回归。

### P1 — 生成器产物质量问题（已修复 ✅）

- `publisher.name` 原误用整条页面标题 → 改为 `extractOrgName()` 从正文抽取公司名，正岛现输出 `青岛正岛食品科技有限责任公司`。
- `keywords` 原碎片化（含「制品」等 2 字碎块）→ 新增子串折叠，剔除被更长词条包含的碎片，现输出干净实体短语。

### P2 — 规则改写引擎缺陷（待模块侧修复，非阻塞）

规则引擎是「零成本兜底骨架」，其缺陷不影响 AI 改写落地，但建议后续修复：
- 定义抽取正则把地名「即墨区」的接续字「即」误判为释义标记（`即/是/指/：`）→ 应在定义抽取时排除地名接续场景。
- FAQ 把整句标题当问句 → 应基于真实问句模式（含「？/如何/怎么/是否」等）生成。
- TL;DR 与首句重复 → 应做压缩提炼而非复制首句。
